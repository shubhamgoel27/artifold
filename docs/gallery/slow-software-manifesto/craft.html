The Slow Software Manifesto

Twelve points, signed by people who would rather own their tools than rent them.

Manifest first published 27 May 2026. Version 1.0. Read the signers. Read the points.

Preface

Software has been getting faster at producing, and slower at serving. It ships every Tuesday and forgets you every Wednesday. It rents you back the file you wrote. It updates itself in the night and renames your menus by morning. It treats your attention as inventory and your patience as runway.

We refuse. Below are the twelve points of slow software. None of them are difficult. All of them are unfashionable.

This document is not a product roadmap. It is a line in the sand. If you agree, sign your name to it. If you disagree, write your own. If you find it obvious, good. The obvious has been forgotten on purpose.

The Twelve Points

  1. 1. Respect the user, or get out of the way.

    The user is the reason the program exists. Not the metric. Not the dashboard. Not the quarterly review. Software that interrupts, nags, upsells, or guilts is software that has forgotten its job. Build a tool. Then leave the room.

  2. 2. Prefer files to apps.

    A file is a noun the user can hold. An app is a verb the company controls. Files can be moved, copied, emailed, backed up, opened in 2050, and read by a program nobody has written yet. Apps can be deprecated on a Thursday. Files outlive apps. Design accordingly.

  3. 3. Prefer plain text to proprietary formats.

    Plain text is the only format with a hundred-year warranty. It opens in anything. It diffs cleanly. It survives the company, the founder, the acquirer, and the bankruptcy. If your format requires the vendor's binary to be readable, your format is a leash. Write to disk in something a human could squint at and understand.

  4. 4. No subscriptions for things that do not change.

    A text editor is not a streaming service. A calculator is not a magazine. Charging rent for a program that ran the same yesterday is a confession that you have nothing new to offer. Sell the thing. Let the buyer keep the thing. Rent only what genuinely costs you every month.

  5. 5. Run locally.

    The fastest network is no network. The most private server is no server. Software that requires a round trip to a faraway machine to flip a checkbox is software that has chosen its convenience over yours. Run on the user's hardware. The hardware is good now. Use it.

  6. 6. Ship infrequently.

    Every release is a small tax on someone's muscle memory. Every redesign asks the world to relearn what already worked. Ship when you have something to say. Then be quiet. A program that has not changed in three years is not abandoned. It is finished.

  7. 7. Embrace constraints.

    A small program does a few things well and refuses the rest. A large program does a thousand things badly and calls it a platform. Choose your scope and defend it. The feature you turn down is the feature that does not break next year. Scope is a moral position.

  8. 8. Be small enough to read.

    The user cannot trust what the user cannot inspect. A program of a few thousand lines can be read by one curious person in one afternoon. A program of a few million lines is a religion: you must take it on faith. Slow software is small software. Small software is honest software.

  9. 9. Do not autoupdate without consent.

    The program on the user's disk belongs to the user. It does not have permission to rewrite itself overnight. Ask before you change. Change only what was asked. The user who said no last Tuesday is allowed to say no this Tuesday. Consent is not a checkbox buried in settings.

  10. 10. Owned, not rented.

    A tool you cannot keep is not a tool. It is an arrangement. When the company turns off the lights, your work should still open. When your card expires, your archive should still read. Buy software the way you buy a hammer: once, fully, forever. The hammer does not phone home.

  11. 11. Archive well.

    Future you is a stranger with no password manager. Write data in formats that stranger can open. Keep names that stranger can search. Avoid clever schemas that only the current version of the program understands. The archive is the product. Everything else is a viewer.

  12. 12. Outlive the company.

    Companies die. Founders leave. Investors lose patience. Servers get shut off in alphabetical order. Software that requires its maker to stay alive is software with an expiration date stamped on the user. Build tools that work after you are gone. That is the only software worth signing your name to.

What we are against

Your software watches you. Your data is collateral. Your file is a row in someone else's table. Your subscription is a hostage negotiation conducted monthly. Your update broke the workflow you spent a year building. Your vendor was acquired and your archive is now read-only and your read-only access expires in ninety days.

None of this is necessary. All of it is a choice. Other choices exist.

We are against the redesign that solved no problem. The dialog box that asked nothing. The notification that meant nothing. The onboarding flow that delayed the work. The empty state that suggested a tutorial instead of a button. The settings panel reorganized by a designer who did not use the product. The migration to the cloud that was a migration of ownership.

What we are for

Programs that start fast and finish. Files on disks you own. Formats older than the people using them. Tools that did not need a relaunch this year because they were correct the first time. Software you can read, fork, and bury in a time capsule with reasonable confidence it will compile when dug up.

We are for the menu that has stayed in the same place since 2008. The keyboard shortcut that still works. The export button that produces something the program did not invent. The save file that is a directory of text files. The program that has no telemetry because it does not need to know.

This is not nostalgia. The 1990s shipped plenty of bad software. We do not want the past back. We want the discipline back.

Frequently asked accusations

You are romanticizing the past.

No. The past had Clippy and Real Player. We are romanticizing the constraints that produced the good parts of the past, not the parts that crashed your computer at the photo plant. Keep the discipline. Drop the bugs.

This will not scale.

Correct. Some things should not. A diary does not scale. A pocket knife does not scale. A favorite restaurant does not scale. Software is allowed to be a favorite restaurant.

Users want convenience.

Users want their work to still exist tomorrow. The convenience that costs them their archive is not convenience. It is debt with a friendly interface.

How will anyone get paid?

By selling the program. By supporting it. By being hired for knowing it. By teaching it. By writing the next one. People got paid for software long before subscriptions existed. They can do it again.

What about security updates?

Ship them. Announce them. Let the user install them. A security patch is not a redesign and should not arrive bundled with one.

What about AI?

The same rules apply. Run the model on the user's machine when possible. Save outputs as plain text. Do not condition the tool on a remote service that may be deprecated. The novelty of the technology does not exempt it from the obligations of the craft.

A short test

Before shipping, ask the following questions. If the answer to any of them is no, the work is not finished.

  • If the company vanishes tomorrow, can the user still open their files?
  • If the network is down, can the user still do the work?
  • If the user does not pay next month, does the archive remain readable?
  • Can the user export everything in a format another program can read?
  • Has any update in the last year removed a feature someone relied on?
  • Does the program ask before changing itself?
  • Could a curious person read the source in an afternoon?
  • Is the file format documented somewhere outside the source code?

These are not aspirational questions. They are minimums. Treat them as such.
