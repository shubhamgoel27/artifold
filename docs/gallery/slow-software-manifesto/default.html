A Manifesto for Slow, File-Based, Anti-Bloat Software

In defense of tools that load fast, store plainly, and respect your time.

Software has gotten heavy. It syncs constantly, demands accounts, hides your data behind APIs, and ships a browser inside a browser. This is a manifesto for the other path: small programs, plain files, and the patience to build things that last.

1.Your data belongs in a file.

A file you can see, copy, email, back up, and open in twenty years. Not a row in someone's database. Not an opaque blob synced to a cloud. If the company disappears tomorrow, your work should still exist on your disk.

2.Plain text outlasts everything.

Markdown, CSV, JSON, YAML, SQLite. These formats have outlived a thousand proprietary apps and will outlive a thousand more. Prefer them. They are readable by humans, by scripts, and by tools that haven't been invented yet.

3.Local first, network second.

An app should work on a plane. It should work when the wifi is bad. It should work when the vendor's servers go down. Cloud sync is a feature, not a foundation. Build for the offline case and the online case becomes free.

4.Startup time is a moral issue.

Every second of cold-start latency is a second stolen from your user. A note-taking app should open faster than you can blink. If your launcher needs a splash screen, you have failed. Respect the clock.

5.Do one thing. Compose the rest.

The Unix philosophy still works. Small tools that do one job well, joined together by pipes and files, beat monolithic suites every time. Stop trying to be a platform. Be a verb.

6.No account required.

Why does a text editor need to know my email address? Why does a calculator want me to "sign in to sync"? Defaults matter. If your software demands an identity before it draws a single pixel, it is not serving the user — it is serving the funnel.

7.Ship in megabytes, not gigabytes.

A chat app does not need 400MB of disk. A music player does not need a Chromium runtime. The bloat is not a law of nature. It is a choice — usually one made for the developer's convenience at the user's expense.

8.Configuration is a text file.

Settings should live in a file you can read, version-control, copy between machines, and edit with any tool. GUI preference panes are fine. Hidden binary preference blobs are not. Configuration is documentation.

9.Backwards compatibility is a feature.

Files written by version 1 should open in version 10. The user did not consent to a rewrite. If you must change formats, provide migration tools and keep the old reader around. Forever is a long time, but it is the right target.

10.Updates are opt-in, not ambushes.

The software the user installed yesterday should be the software they use today. Auto-updates that move buttons, remove features, or change defaults overnight are a betrayal of trust. Stability is a deliverable.

11.The filesystem is the API.

If your app stores its data as files in a folder, every other tool on the system can read, search, sync, and back it up for free. You do not need to build an integration with Dropbox. Dropbox already exists. So does grep.

12.Slow software is patient software.

Slow does not mean sluggish. It means deliberate. It means refusing to chase the next framework, the next paradigm, the next funding round. It means building something today that you will be proud of in ten years — and that your users will still be using.
